# Full Read/Write GitHub Integration — Implementation Plan

## Goal
Make the GitHub Telemetry card a genuine, native GitHub integration: rich read views (PRs, issues, commits, workflow history, branches, checks) PLUS write actions (merge/approve PR, comment, trigger workflow, create release) — gated by a UI confirm for destructive ops, refreshed by polling. Consolidate the scattered GitHub code (the `github.py` router + the `git_github_manager` skill, which duplicate the client and have a token-resolution bug) into one shared client.

## Decisions (confirmed)
- **Approval:** UI confirm modal for destructive writes only (`Alpine.store('modal').confirm(...)`). Read actions need no approval. All logged to audit.
- **Realtime:** polling (extend the existing 10s interval). No webhook ingress now; backend designed so webhooks can be added later.
- **Auth:** reuse the existing `KAZMA_SECRET` middleware — `/api/github/*` is already a sensitive prefix, so all new sub-routes are auto-gated by cookie/header.

## Architecture: 3 layers

### Layer 1 — Shared `GitHubClient` (new, the foundation)
**New file:** `kazma-gateway/kazma_gateway/routers/github_client.py`

A reusable async client that both the router and the skill will use. Eliminates the copy-paste `httpx.AsyncClient` blocks and fixes the skill's token bug.

- `get_github_token()` — extracted from `github.py:157-166`: ConfigStore `connectors.github.token` → `$GITHUB_TOKEN` → `$GITHUB_PAT`. Single source of truth (fixes skill's env-only bug).
- `resolve_repo(cwd) -> (owner, repo)` — extracted from `github.py:107-153` + `parse_github_slug`.
- `GitHubClient` class: base URL `https://api.github.com`, standard headers (`Accept`, `User-Agent`, `Authorization: Bearer {token}` — switch from deprecated `token` to `Bearer`), configurable timeout, **shared rate-limit handling** (`X-RateLimit-Remaining`/`X-RateLimit-Reset` on ALL responses, not just the repo call), `Retry-After`, error mapping → a `GitHubError(status, message, rate_limited)`.
- `get(path, params)`, `post(path, json)`, `patch`, `put`, `delete` — thin wrappers returning parsed JSON or raising `GitHubError`.
- `paginate(path, params, per_page=30, max_pages=10)` — generic **Link-header following** (currently only the count hack exists). Cap pages to bound cost.
- `__aenter__/__aexit__` context-manager over a single `httpx.AsyncClient`.

### Layer 2 — Extended GitHub router (backend write/read endpoints)
**File:** `kazma-gateway/kazma_gateway/routers/github.py` (extend existing)

Refactor the existing `GET /status` and `POST /token` to use `GitHubClient`, then ADD endpoints. All use `JSONResponse` + Pydantic models + the bookmarks.py factory/error conventions. Group with inline section comments.

**NEW read endpoints** (all `GET`, return lists/objects, no approval):
- `GET /api/github/pulls?state=open&limit=20` — PR list with number, title, author, branch, draft, mergeable_state, status-check summary, review states. **GraphQL query** (one round-trip: PR + reviews + checks + head commit), falling back to REST if GraphQL fails.
- `GET /api/github/pulls/{number}` — single PR detail (files, checks, reviews, comments).
- `GET /api/github/issues?state=open&limit=30` — issue list (number, title, labels, author).
- `GET /api/github/commits?limit=20` — recent commits (sha, message, author, date).
- `GET /api/github/workflows?limit=10` — workflow run **history** (not just latest); each with name/status/conclusion/branch/event/url.
- `GET /api/github/branches` — branch list with ahead/behind vs default.
- `GET /api/github/releases?limit=10` — releases (tag, name, prerelease, url).
- `GET /api/github/commits/{sha}/checks` — CI check-runs for a commit (used from PR detail).

**NEW write endpoints** (destructive → return `202` describing the pending action; a confirm is required from the UI before the UI calls the matching execute endpoint — see Layer 3). Each logs `AuditLogger.log_access_attempt` on request and `log_authorization_decision` on confirm:
- `POST /api/github/pulls/{number}/merge` — merge a PR. Body: `{merge_method: "merge"|"squash"|"rebase"}`. UI confirms first.
- `POST /api/github/pulls/{number}/reviews` — submit a review (approve/request-changes/comment). Body: `{event, body}`.
- `POST /api/github/pulls/{number}/comments` — comment on a PR. Body: `{body}`.
- `POST /api/github/pulls/{number}/labels` — add labels. Body: `{labels: [...]}`.
- `POST /api/github/issues/{number}/comments` — comment on an issue.
- `POST /api/github/issues/{number}/labels` — label an issue.
- `POST /api/github/actions/{workflow_id}/dispatches` — trigger a workflow (`workflow_dispatch`). Body: `{ref, inputs}`.
- `POST /api/github/releases` — create a release. Body: `{tag_name, name, body, target_commitish, prerelease}`.
- `POST /api/github/releases/{release_id}` — edit a release (e.g. publish a draft).
- `DELETE /api/github/releases/{release_id}` — delete a release.

Design note: since you chose "UI confirm for writes" (not a pending-action store), the write endpoints **execute immediately** when called — the "approval" is the UI confirm modal gating the call client-side, plus the `KAZMA_SECRET` auth server-side, plus audit logging. This keeps the backend simple (no pending-action DB) while still preventing accidental clicks. All write actions are audit-logged with the acting identity and result.

**Fixes folded in during refactor:**
- Remove hardcoded `g:/GitHubRepos/kazma/.env` fallback in `save_token_to_env` → use `Path.cwd()/.env` only, log if missing.
- Unify `parse_github_slug` + cwd resolution into the client so router and skill share one path.
- Fix 403 on pulls/workflows responses being silently swallowed (apply rate-limit check to all calls).
- Switch auth header to `Bearer`.

### Layer 3 — Frontend (workspace.html GitHub card)
**File:** `kazma-ui/kazma_ui/templates/workspace.html` (extend the existing card)

- **Expand the card** into tabbed sub-views: **Overview** (current stats + latest workflow, keep), **Pull Requests** (list → click to expand detail with checks/reviews + action buttons), **Issues** (list), **Commits** (list), **Workflows** (history list), **Releases** (list). Use the existing `x-for` row + empty-state pattern; tabs via a local `x-data` `activeTab` state.
- **New Alpine state:** `ghPrs []`, `ghIssues []`, `ghCommits []`, `ghWorkflows []`, `ghBranches []`, `ghReleases []`, `ghSelectedPr null`, `ghActiveTab 'overview'`, plus per-action loading flags.
- **New fetch methods** mirroring `loadGitHubStatus()`: `loadGitHubPrs()`, `loadGitHubIssues()`, etc. — called on tab activation + included in the existing 10s `setInterval` (staggered so the GitHub API isn't hammered every 10s — each list on its own lighter cadence, e.g. PRs/issues every 30s, commits/workflows every 60s, overview every 10s).
- **PR detail + actions:** click a PR → `ghSelectedPr` populated from `/api/github/pulls/{n}`; expand panel shows files/checks/reviews and action buttons (Merge [with method dropdown], Approve, Comment) using the swarm.html worker-row button-cluster style (`btn-success` approve, `btn-danger` destructive).
- **Confirm-before-destructive:** use the existing `Alpine.store('modal').confirm(title, message, onConfirm)` (XSS-safe, styled). Add a tiny extension to `app.js` `confirm()` to accept a `confirmVariant` so Merge/Release get a red `btn-danger` confirm button.
- **Action methods:** `mergePr(num, method)`, `submitReview(num, event, body)`, `commentPr(num, body)`, `createRelease(...)`, `triggerWorkflow(id, ref, inputs)` — each: show confirm → `fetch(POST ...)` → toast success/error (`KazmaStream.toast`) → refresh the relevant list. No auth header needed (cookie-based, like all existing fetches).
- **New modals** (global `showModal` pattern) for: Create Release (tag/name/body form), Trigger Workflow (ref/inputs form), New PR comment.

## GraphQL usage
GraphQL is used **only** for the PR list + PR detail endpoints (the nested case: PR + its reviews + its checks + its head commit in one query) — this is where it earns its keep by avoiding 4-5 REST round-trips. All other endpoints use REST. The `GitHubClient` gets a `graphql(query, variables)` method. GraphQL failures fall back to REST composition.

## Out of scope (explicitly deferred)

- A pending-action approval store / HITL gating for these routes (you chose UI confirm).
- Exposing the write ops as agent/swarm tools (would add them to `_EXTENDED_DANGER`); this is UI/API-only for now.
