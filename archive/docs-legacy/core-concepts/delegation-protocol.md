# Delegation Protocol

Kazma supports multi-agent delegation where the orchestrator dispatches tasks to specialized workers via the WorkerRegistry and SwarmEngine.

## Smart-Fallback Routing

When a task is received, the CapabilityRouter tries multiple strategies in sequence:

```
Task: "fix auth bug"
  │
  ├─► 1. Semantic routing (ChromaDB embeddings + all-MiniLM-L6-v2)
  │      → No specialist for "auth"
  │
  ├─► 2. Keyword matching (tag overlap)
  │      → No expertise tag matches
  │
  ├─► 3. Generalist fallback
  │      → Finds "bridge" (no expertise, has Soul) → dispatches ✅
  │
  └─► 4. Last resort — ALL enabled workers
```

Workers without expertise tags or a system_prompt (Soul) are automatically treated as Generalists. This guarantees **zero dispatch failures**.

## Pipeline Topology

```
Researcher → Refiner → Builder → Validator
```

Each stage receives the previous stage's output as context. The Refiner (Middleman) captures all outputs, strips noise, and synthesizes a clean Markdown report card:

```markdown
## Swarm Report

**Task:** fix auth bug
**Status:** ✅ Completed
**Duration:** 2340ms

### Stage Outputs
✅ **researcher** (890ms):
> Authentication flow analyzed...

### Synthesized Summary
> Authentication flow analyzed. Implemented PBKDF2 hashing...

*Generated by Kazma Swarm — 4 stage(s), 4 completed*
```

After completion, the Self-Improvement Engine analyzes the results and may update the worker's system_prompt (Soul) based on success/failure patterns.

## Unified SQLite Logging

Every pipeline step, tool execution, and worker output is persisted to `pipeline_logs.db`:

```python
from kazma_core.swarm.memory.pipeline_logger import get_pipeline_logger

plog = get_pipeline_logger()
plog.log_step(cid, "core", "researcher", "info", "step_start", "Starting analysis")
plog.log_tool_exec(cid, "core", "shell_exec", "git log", "commit1\ncommit2")
plog.log_output(cid, "core", "researcher", "Analysis complete: ...")

# Query from Web UI
logs = plog.query_by_correlation(correlation_id)
```

## WorkerRegistry API

All worker CRUD operations persist to `swarm_registry.json` and survive reboots:

| Operation | REST API | CLI |
|---|---|---|
| Create | `POST /api/swarm/workers` | `kazma swarm worker add` |
| Read | `GET /api/swarm/workers` | `kazma swarm workers` |
| Update | `PUT /api/swarm/workers/{name}` | — |
| Delete | `DELETE /api/swarm/workers/{name}` | `kazma swarm worker remove` |

Workers can be In-Process (runs inside Kazma) or Telegram (dispatches via bot).
